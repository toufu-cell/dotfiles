# search-first - 詳細手順

## 概要

コードを書く前に既存ツール・ライブラリ・実装パターンを調査する「調査先行」ワークフロー。
5フェーズ（ニーズ分析 → 並列検索 → 評価 → 判断 → 実装）で処理する。

---

## 非適用条件（誤発動抑止）

以下に該当する場合はこのスキルを使わない:

- 小さなバグ修正、既存パターンへの追従
- docs / test のみの変更
- 単純リファクタリング
- ユーザーが使用ツール・ライブラリを明示済みの場合

---

## Phase 1: ニーズ分析

ユーザーの要件から以下を明確にする:

1. **必要な機能**: 何を実現したいか（1文で）
2. **言語/フレームワーク制約**: 使用言語、フレームワーク、既存の技術スタック
3. **品質要件**: パフォーマンス、セキュリティ、ライセンス等の制約

---

## Phase 2: 並列検索（段階的に拡大）

### Quick Mode（インライン）

単一エージェントで以下のチェックリストを順次実行:

0. **リポジトリ内検索**: `Grep` で既存モジュール・テスト・類似実装を確認
1. **MCP / スキル検索**: `~/.claude/settings.json` の MCP サーバー一覧と `~/.claude/skills/` を確認
2. **パッケージ検索**: npm / PyPI / Homebrew 等のレジストリを WebSearch で検索
3. **GitHub / Web 検索**: 維持されている OSS 実装を WebSearch で探す

### Full Mode（エージェント）

非自明な機能の場合、Agent ツールでリサーチャーエージェントを起動し並列検索:

```
Agent(subagent_type="general-purpose", prompt="
  以下の機能の既存実装を調査してください:
  機能: [DESCRIPTION]
  言語/フレームワーク: [LANG]
  制約: [ANY]

  検索対象: npm/PyPI, MCP サーバー, Claude Code スキル, GitHub
  出力: 構造化された比較表と推奨
")
```

**フォールバック**: Agent ツールが利用不可の場合は Quick Mode で順次実行する。

**ネットワーク不可時**: ローカル検索結果（リポジトリ内 + MCP/スキル）のみで Phase 3 に進む。

### 検索順序の原則

| 段階 | 範囲 | 承認 |
|------|------|------|
| 1. ローカル検索 | リポジトリ内、MCP、スキル | 不要 |
| 2. レジストリ / Web 検索 | npm, PyPI, GitHub, Web | 不要（読み取り専用） |
| 3. 変更系操作 | インストール、設定変更 | **ユーザー承認必須**（Phase 5 に委譲） |

---

## Phase 3: 評価

### 評価軸

候補を以下の軸でスコアリング（各 1-10）:

| 軸 | 確認ポイント |
|----|-------------|
| 機能性 | 要件をどの程度満たすか |
| メンテナンス | 最終リリース日、Issue 対応状況 |
| コミュニティ | Stars、Downloads、Contributors |
| ドキュメント | README、API ドキュメントの充実度 |
| ライセンス | 許容ライセンス（MIT/Apache/BSD/ISC 等）か |
| 依存関係 | 依存の数と深さ、バンドルサイズ |
| ランタイム互換性 | 使用言語・ランタイムのバージョンと互換性があるか |
| セキュリティ | 既知の脆弱性、セキュリティ監査の有無 |

### 優先順位原則

**既存リソースを新規依存より常に優先する**:

```
既存コード > 標準ライブラリ / built-in > 既存依存の活用 > 新規依存の導入 > Build
```

---

## Phase 4: 判断（決定マトリクス）

| シグナル | アクション |
|---------|-----------|
| 既存コード or 標準ライブラリで実現可能 | **Reuse** — 新規依存不要 |
| 完全一致、良メンテナンス、許容ライセンス | **Adopt** — そのまま導入 |
| 部分一致、良い基盤 | **Extend** — 導入 + 薄いラッパー |
| 複数の弱い一致 | **Compose** — 2-3個の小パッケージを組合せ |
| 適切なものなし | **Build** — 調査を踏まえてカスタム実装 |

### 判断結果をユーザーに提示

```
## 調査結果

### 推奨: {Reuse|Adopt|Extend|Compose|Build}

**候補**: {候補名}
**理由**: {なぜこの判断か}
**スコア**: 機能性 {N}/10, メンテナンス {N}/10, ...

### 代替案
- {代替案1}: {概要}
- {代替案2}: {概要}
```

---

## Phase 5: 実装（ユーザー承認ゲート）

判断結果に基づいて実装する:

| アクション | 実施内容 |
|-----------|---------|
| Reuse | 既存コードを活用。新規ファイル不要 |
| Adopt | パッケージインストール + 設定 |
| Extend | パッケージインストール + 薄いラッパー作成 |
| Compose | 複数パッケージインストール + 統合コード |
| Build | カスタムコード作成（調査結果を参考に） |

**変更系操作（インストール、設定変更等）は全てユーザーの明示的な承認を得てから実行する。**

## 検索ショートカット（カテゴリ別）

### 開発ツール
- Linting → `eslint`, `ruff`, `textlint`, `markdownlint`
- Formatting → `prettier`, `black`, `gofmt`
- Testing → `jest`, `pytest`, `go test`
- Pre-commit → `husky`, `lint-staged`, `pre-commit`

### AI/LLM 連携
- Claude SDK → 最新ドキュメントを確認
- プロンプト管理 → MCP サーバーを確認
- ドキュメント処理 → `unstructured`, `pdfplumber`, `mammoth`

### データ & API
- HTTP クライアント → `httpx` (Python), `ky`/`got` (Node)
- バリデーション → `zod` (TS), `pydantic` (Python)
- データベース → まず MCP サーバーを確認

### コンテンツ & 公開
- Markdown 処理 → `remark`, `unified`, `markdown-it`
- 画像最適化 → `sharp`, `imagemin`

---

## アンチパターン

| パターン | 問題 |
|---------|------|
| **いきなりコードを書く** | 既存ツールの確認をせずにユーティリティを作成する |
| **MCP を無視** | MCP サーバーが既に提供している機能を見落とす |
| **過度なカスタマイズ** | ライブラリを厚くラップしすぎて本来のメリットを失う |
| **依存関係の肥大化** | 小さな機能のために巨大なパッケージを導入する |

---

## エラーハンドリング

| エラー | 対処 |
|-------|------|
| 検索結果が0件 | 検索キーワードを変えて再検索。それでも0件なら Build を推奨 |
| ネットワーク不可 | ローカル検索結果のみで Phase 3 に進む |
| 候補が多すぎる | 上位3候補に絞ってスコアリング |
| ライセンスが不明 | ユーザーに確認してから判断 |
